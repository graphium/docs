## Overview

Once a playbook is published, Builders can run it immediately on demand, test it safely in dry-run mode before going live, or configure it to run automatically on a schedule. This article covers all three run modes and explains how to read the Run History panel to understand what the AI did during each execution.

---

## Running and Testing a Playbook

For a published playbook, the **Run History** panel on the right side of the editor contains two buttons:

- **Test** (flask icon) — Starts a dry-run test that executes all steps but suppresses external actions. Use this to verify your playbook before running it live.
- **Run** (play icon) — Starts a live run immediately. The playbook executes all steps and takes real actions.

</Frame>

A "Test run started" or confirmation toast appears when the run is queued. The new run appears at the top of the Run History list with a **Pending** status while it executes.

> **Note:** The **Save & Test** button in the Playbook Outline header is only available in edit/draft mode. For a published playbook, use the **Test** and **Run** buttons in the Run History panel instead.

---

## What Happens in Test Mode

Test mode executes all steps but suppresses any real-world side effects:

- **External actions are suppressed.** Any step that would send an email, post to a connection, or write to an external system instead logs: *"TEST MODE: Would have executed [action] on [connection name]."*
- **Vault table writes are suppressed.** No records are created or modified.
- **Approval gates are skipped.** The run does not pause for approval in test mode.
- Test runs appear in the Run History panel with a **Test** badge (yellow) so they are visually distinct from live runs.

> **Note:** Test runs still read from your vault tables and connections. Only write actions and external outputs are suppressed.

## Reading the Run History

The **Run History** panel shows all past and in-progress runs for the current playbook. Each run row shows:

- **Date and time** — When the run started.
- **Status badge** — The current state: **Completed** (green), **Pending** (orange), **Failed** (red), or **Cancelled**.
- **Trigger type** — How the run was started: **Manual** (plain text), **Test** (yellow badge), **Scheduled**, **Record Added**, **Connection Event**, etc.
- **Step progress** — How many steps completed out of the total (for example, *2/2 steps*).
- **Execution time** — How long the run took (for example, *20s*).

## Reading a Run Detail

Click any run row to expand its detail inline within the Run History panel. The detail view shows:

- **← All Runs** — Click to return to the run list.
- **"Playbook triggered by [trigger type]"** — Confirms how the run was started.
- **Numbered step tabs** — Click each step number to navigate between steps.
- **AI narration per step** — A plain-language summary of what the AI did at each step, including any tool calls made and their results.

If a step **fails**, the run stops at that step and all subsequent steps are marked as **Skipped**. The error message is shown in the failed step's narration.

## Cancelling a Run

To cancel an in-progress run, open the run detail while the run is still executing and click **Cancel Run**. Steps that have already completed are not rolled back.

## Running a Playbook on a Schedule

Scheduled playbooks run automatically at a configured time and frequency — no manual trigger needed.

### Configuring a Schedule

In the playbook editor, click the trigger card in the Playbook Outline and select **Scheduled** from the Choose a Trigger modal. The Trigger Configuration panel shows:

- **Frequency** — How often the playbook runs (for example, Daily, Weekly).
- **Time** — The time of day the run should start (for example, 09:00 AM).
- **Time Zone** — The time zone used to evaluate the scheduled time.

Save and publish the playbook to activate the schedule. The first scheduled run fires at the next occurrence of your configured time.

<Frame>
  <img src="/images/graphium/playbooks/scheduled-trigger-config.png" alt="Scheduled trigger configuration showing Frequency, Time, and Time Zone fields" />
</Frame>

### Scheduled Run Behaviour

- Scheduled runs appear in the Run History panel with a **Scheduled** trigger type.
- If the playbook is **disabled**, scheduled runs will not fire until the playbook is re-enabled.
- Multiple scheduled playbooks in the same vault can run concurrently without blocking each other.

> 💡 **Tip:** Use test mode to verify your playbook's logic before publishing on a schedule. This prevents a misconfigured playbook from running live on a recurring basis.

## Run Status Reference

| Status | Meaning |
|--------|---------|
| **Pending** | The run is queued and executing |
| **Completed** | All steps finished successfully |
| **Failed** | A step encountered an error and the run stopped |
| **Cancelled** | The run was manually cancelled by a Builder |

## Troubleshooting

| Issue | Likely Cause | What to Do |
|-------|--------------|------------|
| Test and Run buttons are greyed out | The playbook may not be published | Publish the playbook first — the Test and Run buttons are only active for published playbooks |
| Run does not appear in history | A connection or network issue may have interrupted the request | Refresh the page and check again |
| A step shows as Skipped | A previous step failed and the run stopped | Open the failed step's detail to read the AI narration, then correct the step's instructions or resource access |
| Scheduled playbook is not running | The playbook is disabled, or the schedule is not yet published | Confirm the playbook is enabled and that the scheduled trigger has been saved and published |
